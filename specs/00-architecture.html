1 Positionnement de SOLUVIA

SOLUVIA est la couche de pilotage strategique qui se superpose a l'ecosysteme operationnel existant.

Les 3 systemes

E

Eduvia

Role : Outil operationnel
Gere : Projets, contrats, apprenants, financements OPCO, taches qualite, suivi pedagogique, facturation OPCO
Statut : Source de verite pour les donnees metier
Connexion : API REST (lecture)

S

SOLUVIA

Role : Outil de pilotage
Gere : Vision consolidee projets, suivi temps, metriques, finance, facturation interne, performance commerciale
Statut : Consomme Eduvia + produit ses propres donnees
Connexion : API Eduvia (lecture) + API Odoo (ecriture)

O

Odoo

Role : Comptabilite
Gere : Plan comptable, ecritures, TVA, exports legaux
Statut : Recepteur des factures generees par SOLUVIA
Connexion : API XML-RPC / REST (ecriture factures + lecture paiements)

⚠

Principe fondamental — SOLUVIA ne remplace jamais Eduvia. Les donnees operationnelles (contrats, apprenants, taches qualite) restent gerees dans Eduvia. SOLUVIA les lit via API et y ajoute sa couche de pilotage (temps, metriques calculees, facturation interne, performance commerciale).

Design login — Page plein ecran, fond gris clair, carte centree avec logo SOLUVIA. Pas de sidebar, pas de topbar. Le route group (auth) utilise un layout vide.

Comportement apres connexion

  • Connexion reussie → Redirection vers /projets (page d'accueil)
  • Email inexistant → Message : "Identifiants incorrects" (pas de leak d'information)
  • Mot de passe faux → Meme message generique
  • Compte desactive → Message : "Votre compte a ete desactive. Contactez un administrateur."
  • Mot de passe oublie → Saisir email, reception d'un lien magic link, redirection vers formulaire de nouveau mot de passe
  • Rate limiting → 5 tentatives max, puis blocage 15 minutes (Supabase Auth natif)

4 Roles & Permissions

Deux roles : Admin (acces total) et Chef de Projet / CDP (acces restreint a ses projets).

Definition des roles

A

Admin

Acces : Toutes les pages, tous les projets, toutes les actions
Peut : Creer/modifier des projets, gerer les users, configurer les APIs, generer des factures, voir la performance commerciale
Nombre estime : 2-3 personnes

C

Chef de Projet (CDP)

Acces : Pages limitees, uniquement ses projets assignes (cdp_id ou backup_cdp_id)
Peut : Saisir son temps, voir ses projets/qualite/finance, supprimer des lignes de facturation
Ne peut pas : Voir les autres projets, acceder a l'admin, la perf commerciale, les parametres
Nombre estime : 25-30 personnes

Implementation RLS (Row Level Security)

-- Politique pour la table projets
CREATE POLICY "admin_all" ON projets
  FOR ALL
  USING (get_user_role() = 'admin');

CREATE POLICY "cdp_own_projects" ON projets
  FOR SELECT
  USING (
    cdp_id = auth.uid()
    OR backup_cdp_id = auth.uid()
  );

-- Politique pour saisies_temps
CREATE POLICY "cdp_own_time" ON saisies_temps
  FOR ALL
  USING (user_id = auth.uid());

📌

Implementation — Les notifications sont stockees dans une table notifications (Supabase). Le panneau utilise Supabase Realtime pour afficher les nouvelles notifs en temps reel sans refresh. Les notifications lues sont marquees avec un timestamp read_at.

Table notifications

notifications
  id UUID PK
  user_id FK → users   -- destinataire
  type TEXT               -- facture_retard, tache_retard, rappel_temps, etc.
  titre TEXT
  message TEXT
  lien TEXT               -- URL de redirection au clic
  read_at TIMESTAMPTZ     -- NULL si non lue
  created_at TIMESTAMPTZ
