1 Principes

Page /facturation — seule page de SOLUVIA ou le CDP cree des documents financiers.

Echeances automatiques

RegleDetail
ModeleDecalage de 2 mois. Le mois 1 du contrat est facture fin M+2 (echeance fin M+3). Mensuel jusqu'a M+10. Derniere facture a M12 couvrant les mois 10, 11, 12 (3/12). Total = 10 factures par contrat.
GenerationSOLUVIA calcule les echeances atteintes (date du jour ≥ date d'emission prevue) et les propose au CDP.
MontantCommission SOLUVIA = montant contrat OPCO × taux commission projet ÷ 12 (ou × 3/12 pour la derniere).
Regroupement1 facture = 1 projet × 1 mois. Tous les contrats du projet sont regroupes comme lignes de la meme facture.

6 Avoirs

Document d'annulation totale ou partielle d'une facture. Meme sequence de numerotation.

RegleDetail
DeclenchementDepuis la modale d'une facture, bouton "Emettre un avoir". CDP ou admin.
MotifsRupture anticipee de contrat, erreur de facturation, remise commerciale, ajustement. Motif obligatoire (champ texte).
NumerotationMeme sequence FAC-TRI-NNNN. L'avoir prend le numero suivant dans la sequence globale.
MontantNegatif. Avoir total = montant de la facture en negatif. Avoir partiel = montant saisi par le CDP.
RuptureCalcul automatique au pro-rata temporis : montant contrat × duree realisee / duree totale. Le CDP peut ajuster.
PDFMeme template que la facture avec mention "AVOIR" en rouge et reference a la facture d'origine.
OdooL'avoir est pousse vers Odoo comme la facture (ecriture comptable d'annulation).
⛔

Pas de suppression — Une facture emise ne peut jamais etre supprimee (obligation legale). Pour annuler, on emet un avoir. La facture d'origine reste visible avec le badge Avoir et un lien vers l'avoir correspondant.

📌

Avoir emis par erreur — Un avoir ne peut pas etre supprime non plus. Si un avoir a ete emis par erreur, on emet une nouvelle facture (dans la meme sequence FAC-TRI-NNNN) pour compenser l'avoir errone. Le motif doit mentionner "Compensation avoir FAC-TRI-NNNN emis par erreur".

8 Integration Odoo

Odoo n'est pas encore en production. Cette section decrit le flux prevu et les endpoints a implementer.

Flux SOLUVIA → Odoo (push facture)

EtapeDetail
DeclencheurA la validation d'une facture par le CDP (etape 3 du flux).
Donnees envoyeesN° facture, date emission, date echeance, client (SIRET), lignes de detail (contrat, montant HT), TVA, total TTC.
Action OdooCreation d'une ecriture comptable (facture client) dans le journal de vente.
RetourOdoo renvoie un odoo_id stocke dans la table factures pour le suivi.
AvoirsMeme flux. L'avoir est pousse comme une facture d'avoir (montant negatif).
PDFLe PDF genere par SOLUVIA est joint a l'ecriture Odoo comme piece jointe.

Flux Odoo → SOLUVIA (pull paiements)

EtapeDetail
FrequenceCRON Supabase Edge Function — toutes les heures (a ajuster selon le volume).
Donnees recupereesPour chaque facture ayant un odoo_id : statut paiement, montant paye, date paiement, paiement partiel.
Mise a jour SOLUVIATable paiements mise a jour. Statut de la facture passe a "Payee" (si total) ou reste "Emise" (si partiel).
RetardSi echeance depassee et montant non paye → statut passe a "En retard" automatiquement (CRON ou calcul temps reel).

API prevue

DirectionMethodeEndpoint (previsionnel)Format
Push factureXML-RPC ou REST/api/v1/account.move/createJSON : partner_id, invoice_date, invoice_line_ids, ...
Push avoirXML-RPC ou REST/api/v1/account.move/create (type = out_refund)JSON : idem + reference facture d'origine
Pull paiementsXML-RPC ou REST/api/v1/account.payment/search_readFiltre sur invoice_ids, retour : amount, payment_date, state
⚠

Odoo non disponible — En attendant Odoo, l'admin peut saisir manuellement les encaissements (date + montant) depuis la modale facture. Cette saisie manuelle sera remplacee par le pull automatique une fois Odoo connecte.

Mode degrade & gestion d'erreurs

ScenarioComportement
Push facture echoueLa facture est creee dans SOLUVIA (PDF + email envoyes). Le push Odoo est marque en erreur. Retry automatique toutes les heures (max 5 tentatives). Alerte admin apres 5 echecs.
Pull paiements echoueLe CRON log l'erreur. Les statuts de factures restent inchanges. Retry au prochain cycle. Alerte admin si echec > 24h.
Odoo non configurePas de push ni pull. L'admin saisit manuellement les encaissements. La colonne paiements.saisie_manuelle = true permet d'identifier les saisies manuelles a migrer.
LogsToutes les interactions Odoo (push, pull, erreurs) sont tracees dans une table odoo_sync_logs (timestamp, direction, statut, payload, erreur).

10 Permissions & RLS

Meme modele de permissions que le reste de SOLUVIA, avec des specificites pour la facturation.

A

Admin

Voit toutes les factures. Peut facturer hors fenetre sans avertissement. Peut emettre des avoirs. Peut saisir manuellement les encaissements (fallback Odoo).

C

CDP

Voit uniquement les factures de ses projets (cdp_id ou backup_cdp_id). Peut valider les echeances et emettre des avoirs. Avertissement hors fenetre 25-3.
